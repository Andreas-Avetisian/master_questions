# Define basic Bloom filters and discuss usage and characteristics. Give an example for their utilization and how they can enhance performance.

## Definition

A **Bloom filter** is a *probabilistic data structure* used to answer the question *"has this element been seen before?"* It is related to a hash table but is designed to be much more space- and time-efficient, at the cost of allowing a small rate of *false positives*. The lecture summarises it succinctly: a Bloom filter *"works like a 'fingerprint' of an element, but many elements have the same fingerprint."*

**Structure**: typically an array of $k \times n$ bits (or $n_1 + n_2 + n_3$ separate bit-arrays), initialised to zero, plus a selection of **k hash functions**.

**Two operations**:

- **Add(x)**: hash x with all k hash functions and set the corresponding bits in the array to 1.
- **Contains(x)**:
  - *True* means "x **might** be in the filter" (false positives are possible).
  - *False* means "x is **definitely not** in the filter" (no false negatives).

This asymmetry is the defining property: a Bloom filter can rule an element out with certainty, but can only confirm membership with some probability.

## Motivation and usage

The lecture motivates Bloom filters with the general *"problem at hand"*: a **stream of information** (IP addresses, users accessing a file, DNS profiles), where the source data space is sparsely populated and we need to check repeatedly whether an incoming item has already been encountered.

The naive alternatives are expensive:

- An **array** indexed by all possible values gives O(1) lookup, but wastes space (a 128-bit string would require $2^{128}$ entries).
- A **linked list** or search tree of encountered items reduces space to $|E|$ but slows retrieval to linear or logarithmic time.
- A **hash table** stores the items explicitly, so it inherits their size and needs collision handling.

Bloom filters fit exactly the scenario the lecture lists on the motivation slide: *"large set V of possible items, each item is relatively big, only few items are actually encountered, expensive lookups for large tables."* Their use is everywhere an approximate, cheap membership test is good enough: caching (avoid querying the slow backend for keys that definitely do not exist), deduplication on streams, network routers, blacklists, distributed systems, DNA sequence matching, etc.

## Characteristics

- **Probabilistic**: may return false positives, never false negatives.
- **Very space-efficient**: only a bit-array plus k hash functions, independent of the size of the elements themselves.
- **Very time-efficient**: Add and Contains both run in O(k), where k is the number of hash functions, typically small.
- **No deletion in the basic variant**: clearing bits would introduce false negatives (counting Bloom filters exist as an extension, but are outside the basic definition).
- **No cryptographic requirement** on the hash functions, only good, uniform distribution.
- **False-positive rate** depends on the size of the filter, the size of |E| (encountered elements), and how evenly the hash functions distribute elements across the filter.

## Example from the lecture

The deck walks through a toy example with **patient data** and three simple hash functions:

1. number of words in the sentence mod 10,
2. length of text mod 10,
3. number of patient IDs mod 10.

Inserting two records:

- *"Patient 456 is fine."* hashes to (4, 0, 1).
- *"Patient 764 has a cold"* hashes to (5, 2, 1).

The bit array now has ones at positions 4, 0, 1, 5, 2 (position 1 shared).

Now query *"Patient 4345 is unwell"*. Its hashes are (4, 2, 1), all three bits are already set (by the two earlier records, not by this one), so Contains returns True even though this record was never inserted. This is exactly a **false positive**, and the lecture labels it as such on the example slide.

## How they enhance performance

The closing slide frames the trade-off: *"Why accept false positives? Speed high, space requirements low."* A Bloom filter sits *in front of* an expensive lookup and answers the overwhelming majority of negative queries instantly and in a few bytes of memory. Only the small fraction of queries that pass the filter (true positives plus a tuneable rate of false positives) need to touch the slow, exact data structure. This turns a hot path from O(|E|) or O(log |E|) plus heavy memory footprint into O(k) bit operations on a tiny array, which is what makes Bloom filters a classic performance optimisation in caches, databases, and network middleware.

## Short summary

A **Bloom filter** is a compact, probabilistic membership data structure: a bit array plus k hash functions, with two operations Add and Contains. Contains returns *False* with certainty but *True* only probabilistically, so false positives are possible while false negatives are not. Their characteristics, small memory, constant-time operations, no need for cryptographic hashes, make them ideal as a cheap pre-filter in front of expensive lookups, as illustrated in the lecture's patient-data example where *"Patient 4345 is unwell"* is mistakenly flagged as present. They enhance performance by trading a small, tuneable false-positive rate for dramatic savings in space and lookup time.
